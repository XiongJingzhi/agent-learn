# Level 1 挑战项目

> **难度**: ⭐⭐⭐⭐⭐ 高级
> **预计时间**: 4-6 小时
> **目标**: 在真实约束下交付可靠的 Agent 系统

---

## 挑战项目概述

本 level 的挑战项目包含三个独立的挑战，每个挑战都模拟真实场景中的约束和问题：

| 挑战 | 主题 | 难度 | 时间 |
|------|------|------|------|
| 挑战 1 | 真实约束下交付 | ⭐⭐⭐⭐ | 1.5-2 小时 |
| 挑战 2 | 故障演练与恢复 | ⭐⭐⭐⭐⭐ | 2-3 小时 |
| 挑战 3 | 同行评审与改进 | ⭐⭐⭐⭐ | 0.5-1 小时 |

## 挑战 1：真实约束下交付

### 场景描述

你正在为一位客户开发一个 Agent 系统，客户给出了严格的约束条件：

**时间约束**：
- 必须在 1.5 小时内完成最小可验收版本（MVP）
- 总开发时间不能超过 3 小时

**质量约束**：
- 代码必须有类型提示
- 代码必须有文档字符串
- 必须通过基本的代码审查

**功能约束**：
- 只实现最核心的功能
- 不做任何扩展功能
- 不做任何优化

### 任务要求

1. **定义范围**（10 分钟）
   - 列出 3 个必须实现的核心功能
   - 列出 3 个明确不做的事情
   - 获得客户（你的导师或朋友）确认

2. **交付 MVP**（1.5 小时）
   - 实现核心功能
   - 添加基本的错误处理
   - 编写简单的文档

3. **风险识别**（20 分钟）
   - 识别至少 3 个潜在风险
   - 为每个风险定义缓解措施

### 交付物

- [ ] 功能范围文档（什么做、什么不做）
- [ ] 可运行的 MVP 代码
- [ ] 基本的文档（README.md）
- [ ] 风险清单和缓解措施

### 评分标准

| 维度 | 权重 | 标准 |
|------|------|------|
| **交付完整度** | 35% | 所有承诺的核心功能都实现了吗？ |
| **风险控制** | 35% | 风险识别是否全面？缓解措施是否有效？ |
| **结果可复现** | 30% | 其他人能否根据文档复现结果？ |

### 示例

**功能范围**：
```
必须做：
1. 能够接收用户输入
2. 能够调用搜索工具
3. 能够返回结果

不做：
1. 不做对话历史
2. 不做错误重试
3. 不做日志记录
```

**风险清单**：
```
风险 1：API 调用失败
缓解措施：捕获异常，返回友好错误信息

风险 2：搜索无结果
缓解措施：返回默认提示，而不是崩溃

风险 3：用户输入无效
缓解措施：校验输入，提示用户重新输入
```

## 挑战 2：故障演练与恢复

### 场景描述

你的 Agent 系统已经上线，现在需要模拟三种故障场景，并演练恢复流程。

### 故障场景

#### 场景 1：API 超时

**故障描述**：
- LLM API 调用超时（>30 秒）
- 用户请求失败

**演练任务**：
1. 识别超时错误
2. 实现超时处理（返回缓存或默认响应）
3. 记录超时事件
4. 配置合理的超时时间

**验收标准**：
- [ ] 能够识别超时错误
- [ ] 有降级方案
- [ ] 有超时日志
- [ ] 超时时间合理（不是无限等待）

---

#### 场景 2：错误数据

**故障描述**：
- 工具返回了格式错误的数据
- Agent 无法解析结果

**演练任务**：
1. 检测数据格式错误
2. 实现数据校验
3. 提供默认值或跳过该数据
4. 记录数据错误

**验收标准**：
- [ ] 能够检测格式错误
- [ ] 有数据校验逻辑
- [ ] 不会因为数据错误崩溃
- [ ] 有错误日志

---

#### 场景 3：依赖不可用

**故障描述**：
- 外部服务（如搜索 API）不可用
- 连接失败或返回 500 错误

**演练任务**：
1. 检测连接错误
2. 实现重试机制（指数退避）
3. 实现降级方案（使用缓存）
4. 配置最大重试次数

**验收标准**：
- [ ] 能够检测连接错误
- [ ] 有重试逻辑
- [ ] 有降级方案
- [ ] 配置了最大重试次数

### 交付物

- [ ] 演练日志（每个场景的执行日志）
- [ ] 恢复报告（恢复步骤和结果）
- [ ] 改进后的代码（包含错误处理）

### 评分标准

| 维度 | 权重 | 标准 |
|------|------|------|
| **故障识别** | 40% | 是否正确识别了所有故障？ |
| **恢复有效性** | 40% | 恢复措施是否有效？ |
| **复盘质量** | 20% | 复盘是否深入？是否有改进建议？ |

## 挑战 3：同行评审与改进

### 场景描述

你刚刚完成了一个 Agent 项目，现在需要邀请一位同行（导师、同学、朋友）进行代码评审，并根据反馈进行改进。

### 任务要求

#### 步骤 1：准备评审材料（15 分钟）

准备以下材料：
- [ ] 项目简介（1 页 PPT 或文档）
- [ ] 核心代码（关键部分）
- [ ] 运行截图或演示
- [ ] 评审清单（希望评审什么）

#### 步骤 2：邀请评审（5 分钟）

- [ ] 邀请一位同行
- [ ] 说明评审范围和时间
- [ ] 提供评审材料

#### 步骤 3：收集反馈（30 分钟）

请评审者关注以下方面：
- [ ] 代码可读性（命名、注释、结构）
- [ ] 代码质量（类型提示、错误处理）
- [ ] 功能完整性（是否满足需求）
- [ ] 潜在问题（bug、性能、安全）

#### 步骤 4：分析反馈（15 分钟）

- [ ] 整理所有反馈意见
- [ ] 分类：必须改、建议改、可选
- [ ] 优先级排序

#### 步骤 5：实施改进（30 分钟）

- [ ] 修复必须改的问题
- [ ] 改进建议改的问题
- [ ] 考虑可选的改进

#### 步骤 6：总结反思（10 分钟）

- [ ] 记录最有价值的 3 条反馈
- [ ] 记录最大的 3 个改进
- [ ] 总结经验教训

### 交付物

- [ ] 评审材料（简介、代码、截图）
- [ ] 评审记录（所有反馈意见）
- [ ] 改进计划（分类和优先级）
- [ ] 改进后的代码
- [ ] 总结反思文档

### 评分标准

| 维度 | 权重 | 标准 |
|------|------|------|
| **反馈吸收程度** | 40% | 是否认真听取并吸收了反馈？ |
| **改进幅度** | 30% | 改进是否明显？ |
| **文档完整性** | 30% | 评审材料是否完整？记录是否详细？ |
